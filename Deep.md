Here's a full breakdown of how btop handles theming with raw ANSI escape codes.

---

## 1. Escape Code Primitives (`Fx` namespace)

All escape sequences start from a single base string `Fx::e = "\x1b["`. Text effects (bold, italic, etc.) are pre-built constants on top of it. btop:64-94 

A special `Fx::reset` string is not a constant — it's a `string` that gets **rebuilt on every theme change** to include the theme's fg/bg colors (see §3 below). btop:79-84 

---

## 2. Color Escape Code Generation

Theme colors are stored as pre-built ANSI escape strings. The two conversion functions are:

**`hex_to_color(hexa, t_to_256, depth)`** — converts a `#rrggbb` or `#gg` (greyscale) hex string into an escape sequence:
- Foreground (`depth == "fg"`): `\x1b[38;2;R;G;Bm` (24-bit) or `\x1b[38;5;Nm` (256-color)
- Background (`depth == "bg"`): `\x1b[48;2;R;G;Bm` or `\x1b[48;5;Nm` btop:153-190 

**`dec_to_color(r, g, b, t_to_256, depth)`** — same but takes decimal RGB directly. btop:192-199 

The `depth` parameter controls whether `38` (fg) or `48` (bg) is used in the SGR sequence. The `lowcolor` config flag switches between 24-bit (`2;`) and 256-color (`5;`) mode.

---

## 3. Theme Storage: Three Maps

After loading, all colors live in three global maps in the `Theme` namespace:

| Map | Type | Contents |
|---|---|---|
| `Theme::colors` | `unordered_map<string, string>` | Named color → ready-to-use escape string |
| `Theme::rgbs` | `unordered_map<string, array<int,3>>` | Named color → `{R, G, B}` decimals |
| `Theme::gradients` | `unordered_map<string, array<string,101>>` | Gradient name → 101 escape strings (indices 0–100) | btop:56-68 

Gradients are generated by `generateGradients()`, which linearly interpolates between `_start`, `_mid`, and `_end` RGB values across 101 steps, then calls `dec_to_color` for each step. btop:290-347 

---

## 4. `setTheme()` — Wiring fg/bg into `Fx::reset`

After generating all colors, `setTheme()` does a critical step: it sets `Term::fg` and `Term::bg` to the theme's `main_fg`/`main_bg` escape strings, then **rebuilds `Fx::reset`** to include them:

```
Fx::reset = "\x1b[0m" + Term::fg + Term::bg
``` btop:445-448 

This means every time `Fx::reset` is emitted anywhere in the draw code, it doesn't just clear styles — it also **restores the theme's base foreground and background colors**.

---

## 5. TTY Mode

For terminals without 24-bit color support, a separate `TTY_theme` map stores raw legacy ANSI codes (e.g. `\x1b[37m`, `\x1b[41m`) directly as strings, bypassing the hex conversion entirely. btop:94-137 

---

## 6. Per-Frame Rendering

The rendering loop runs in a **secondary thread** (`Runner::_runner`). Each frame:

1. `output` string is cleared.
2. Each visible box's `draw()` function is called — these build strings by concatenating `Mv::to(y, x)` cursor moves, `Theme::c("color_name")` escape codes, `Theme::g("gradient_name").at(value)` gradient escapes, `Fx::b`/`Fx::ub` bold toggles, and actual text characters.
3. The full `output` string is written to `cout` in one shot, wrapped in **synchronized output** markers (`\x1b[?2026h` / `\x1b[?2026l`) to prevent tearing. btop:614-660 btop:820-824 

Each `draw()` function ends with `+ Fx::reset` to restore the theme base colors after the box content. btop:1359-1361 

When a menu overlay is active, the main `output` is first stripped of all color codes via `Fx::uncolor()` (regex replace), then the overlay is printed on top. btop:820-823 

---

## Per-Frame Rendering and Terminal Flushing

### Thread Architecture

All rendering runs in a dedicated **secondary thread** (`Runner::_runner`), not the main thread. The main thread handles input events and signals `Runner::run()` to trigger a frame. btop:557-558 

The thread loops on a condition variable (`thread_wait()`), wakes up when triggered, acquires an `atomic_lock`, then does all collect+draw work. btop:577-592 

---

### What Happens Each Frame

**1. Clear the output buffer**

`output` is a plain `std::string` that accumulates the entire frame's content: btop:491-491 btop:614-614 

**2. Collect + Draw each box**

For each enabled box, btop first collects data, then calls the box's `draw()` function which returns a `string`. That string is `+=` appended to `output`:

```
output += Cpu::draw(cpu, gpus_ref, conf.force_redraw, conf.no_update);
output += Mem::draw(mem, conf.force_redraw, conf.no_update);
output += Net::draw(net, conf.force_redraw, conf.no_update);
output += Proc::draw(proc, conf.force_redraw, conf.no_update);
``` btop:660-660 btop:697-697 btop:717-717 btop:737-737 

Each `draw()` function builds its string by concatenating:
- `Mv::to(line, col)` — cursor positioning (`\x1b[line;colf`)
- `Theme::c("color_name")` — a pre-built color escape string
- `Theme::g("gradient")[value]` — one of 101 gradient escape strings
- `Fx::b`, `Fx::ub`, etc. — text effect toggles
- Actual text/box-drawing characters
- `Fx::reset` at the end to restore base fg/bg btop:98-118 

**3. Overlay handling**

If a menu overlay is active, the main `output` is stripped of all color codes via `Fx::uncolor()` (regex replace), dimmed with `Theme::c("inactive_fg")`, and the overlay string is appended on top: btop:820-823 

---

### The Flush

The entire `output` string is written to `cout` in **one single shot**, wrapped in **synchronized output** escape codes to prevent the terminal from rendering partial frames:

```cpp
cout << Term::sync_start
     << output   // (or overlay-composited version)
     << Term::sync_end
     << flush;
``` btop:820-823 

`sync_start` and `sync_end` are DEC private mode sequences:

```
sync_start = "\x1b[?2026h"   // begin synchronized update
sync_end   = "\x1b[?2026l"   // end synchronized update
``` btop:139-140 

The terminal buffers all rendering between these two sequences and only paints to screen when `sync_end` arrives, eliminating tearing. The `flush` forces the OS write buffer out immediately.

---

│
├─ output.clear()

├─ Cpu::collect() → Cpu::draw()  ──► output += "...ESC codes + text..."
├─ Mem::collect() → Mem::draw()  ──► output += "...ESC codes + text..."

```

The key design point: **there is no per-cell or per-line write**. The whole frame is one giant string built in memory, then written to stdout in a single call, gated by synchronized output markers.
